# 参数修改草稿弹窗重构设计

## 背景

当前 `/parameters` 页面里的修改草稿区域仍然沿用右侧抽屉式布局。这个形态在简单参数上还能勉强使用，但一旦遇到 DTS 这类长配置，卡片会被压得很窄，当前值、目标值和修改原因都容易挤在一起，按钮也会看起来像普通文字。用户已经明确希望把它改成和“查看详情”一致的正常弹窗，并且让复杂参数和简单参数都保持良好的可读性。

这次重构只针对参数工作台里的草稿编辑体验，不改现有的参数详情弹窗，不改路由，也不把草稿区继续做成侧边栏。

## 目标

- 把修改草稿从右侧抽屉改成居中的正常弹窗。
- 让弹窗在简单参数和复杂 DTS 参数下都保持良好的视觉层级。
- 让复杂参数的长文本不被强制换行，优先通过更宽的弹窗和代码块横向滚动来保证可读性。
- 让“关闭 / 提交参数 / 全部清空 / 移除本项”等关键操作都呈现为明确的按钮，而不是文本链接。
- 保持现有草稿数据流和提交逻辑不变，只改呈现层和局部交互。

## 非目标

- 不修改 `/parameter-comparison` 的退场状态。
- 不重做参数详情弹窗的跨项目对比能力。
- 不新增新的草稿存储模型。
- 不把这个弹窗改成全屏页面。
- 不影响 `WorkbenchSheet` 在其他页面里的既有用途。

## 设计决策

### 1. 新增页面专用弹窗，而不是改成全局抽屉

参数工作台使用一个新的页面级弹窗组件，例如 `ParameterDraftDialog`。它的视觉语言借鉴 `ParameterDetailDialog`：居中、带遮罩、头部和底部边界清晰、按钮成组排列。

保留 `WorkbenchSheet` 给其他工作流使用，避免把这次改动扩散到节点调试页等无关场景。

### 2. 弹窗采用宽体自适应，而不是固定右栏宽度

弹窗默认宽度按 `min(1120px, calc(100vw - 48px))` 级别渲染；只要当前草稿中存在复杂参数，就切到更宽的模式，例如 `min(1440px, calc(100vw - 48px))`。

弹窗主体内部滚动，卡片列表滚动，外层不再使用右侧抽屉的固定 440px 逻辑。

### 3. 草稿卡片分成“简单参数”和“复杂参数”两种布局

简单参数保持轻量、紧凑、直观。

复杂参数使用更像代码 diff 的布局：当前值和目标值都以代码块方式展示，保留换行、缩进和逗号，不做软换行。编辑区优先用等宽字体和横向滚动，必要时允许单卡横向滚动，而不是把内容压窄。

### 4. 按钮必须像按钮

头部的“全部清空”、卡片里的“移除本项”、底部的“关闭 / 提交参数”都使用明确的按钮样式。禁止再出现看起来像两段文字的底部操作区。

## 信息结构

### 弹窗头部

- 标题：`修改草稿`
- 副标题：`本轮提交 X 项，可先收集多个参数，再统一提交审阅`
- 右侧放关闭按钮，按钮样式与详情弹窗一致

### 弹窗主体

主体分成两个部分：

1. 顶部摘要条
   - 显示当前草稿数量
   - 提供“全部清空”按钮
   - 在复杂参数较多时，摘要条仍然保持单行或两行，不被卡片内容挤压

2. 草稿卡片列表
   - 每个草稿一张卡
   - 卡片顶部显示参数名、模块、风险等级
   - 卡片主体根据参数复杂度切换布局
   - 卡片底部保留移除操作

### 弹窗底部

- 左侧显示本轮草稿的状态说明，例如提交后仍可继续进入“本轮已修改参数表”
- 右侧放两个按钮：`关闭`、`提交参数`
- 当草稿里有空目标值时，`提交参数` 禁用

## 卡片布局

### 简单参数

适用于单行值、短数值、短字符串。

- 顶部一行显示参数名、模块、风险
- 中间用 `当前值 → 目标值` 的方式展示差异
- 目标值使用普通输入框或短文本编辑区
- 修改原因独立成一行输入区
- 卡片高度保持紧凑，避免空白过多

### 复杂参数

适用于 DTS、多行配置、长逗号串、`<>` 包裹的结构化值、或者长度明显较长的参数。

- 卡片顶部增加一个轻量的“结构化参数”标识
- 当前值与目标值采用代码块式 diff 视觉
- 当前值为只读区，目标值为可编辑区
- 保留完整换行和缩进，不做自动软换行
- 代码块区域允许横向滚动
- 目标值编辑区使用等宽字体，行高更疏朗，便于比对
- 修改原因放在 diff 区域之后，避免被长代码挤到首屏之外

复杂参数示例应该能原样容纳类似下面的 DTS 配置：

```text
fast-charge-profile-matrix =
  "0", "5000", "1500", "40", "entry",
  "1", "9000", "3000", "43", "balanced",
  "2", "11000", "4200", "46", "burst";
```

以及使用 `<>` 包裹的结构化写法。

## 响应式规则

- 宽屏下优先保持卡片的两段式结构。
- 中等宽度下，复杂参数卡片可以上下堆叠，但代码块仍然不换行。
- 窄屏下，弹窗保持居中，不退回右侧抽屉；代码块区域允许横向滚动，按钮改为纵向堆叠。
- 不允许底部操作区挤成“纯文字”。

## 交互流程

1. 用户在参数表中点击编辑，草稿进入本轮草稿集合。
2. 草稿弹窗自动打开。
3. 用户在简单参数和复杂参数卡片内分别编辑目标值与原因。
4. 用户可单项移除，也可全部清空。
5. 当所有草稿都有目标值时，提交按钮可用。
6. 提交后草稿进入本轮已修改参数表，弹窗关闭。
7. 如果删除最后一项草稿，弹窗自动关闭。

## 数据与组件边界

### `ParametersPage`

继续作为草稿状态的唯一拥有者，负责：

- `drafts`
- `draftItems`
- `sheetOpen`
- `selectedIds`
- 提交到“本轮已修改参数表”的逻辑

### 新弹窗组件

新增 `ParameterDraftDialog` 或同等级别组件，负责：

- 草稿头部和底部布局
- 简单 / 复杂参数的卡片渲染
- 清空、移除、关闭、提交按钮
- 只做展示和事件回调，不直接改全局状态

## 无障碍

- 弹窗使用 `role="dialog"` 和 `aria-modal="true"`。
- 标题要可被屏幕阅读器识别。
- 关闭、清空、移除、提交都要有明确的可访问名称。
- 复杂代码块在键盘导航下依然可聚焦、可滚动。
- 禁止用纯文本代替按钮。

## 测试

需要补充或更新以下测试：

- 打开草稿后渲染的是居中的弹窗，而不是右侧抽屉。
- 简单参数卡片使用紧凑布局。
- 复杂参数卡片保留换行和缩进，不发生软换行。
- “全部清空 / 移除本项 / 关闭 / 提交参数” 都是可识别按钮。
- 删除最后一条草稿会自动关闭弹窗。
- 提交后草稿进入本轮已修改参数表。
- 在浏览器里验证 `/parameters` 页面在 1440、1280、1024、768 和更窄宽度下都不出现按钮溢出或内容挤压。

## 验收标准

- 草稿区不再是右侧抽屉。
- 弹窗在视觉上和“查看详情”保持同类气质。
- 简单参数看起来简洁，复杂 DTS 参数看起来可读。
- 长配置不会被强制换行到难以辨认。
- 底部操作区是清晰的按钮组，不再像两段文字。
- 现有草稿提交逻辑不变，只改 UI 和局部布局。
