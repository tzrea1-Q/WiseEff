# 参数管理个人工作台设计方案

> **历史文档 / 已被取代：** 本方案已由 [`docs/zh-CN/superpowers/specs/2026-07-07-parameter-home-production-redesign-design.md`](../zh-CN/superpowers/specs/2026-07-07-parameter-home-production-redesign-design.md)（生产化重构，已实施）演进并取代。保留此文档仅作历史设计记录，请以 2026-07-07 方案为当前实现依据。

## 目标

将现有 `/parameter-home` 从“参数管理看板”改造成“个人工作台”。页面不再只回答“参数治理态势如何”，而是优先回答用户登录后的核心问题：“我现在应该做什么？”

新工作台需要保留并合理安排原有看板能力，包括核心指标、参数更新趋势、项目风险分布、热门热区、AI 评分拆解和上下文跳转。区别在于这些看板内容不再是首屏主角，而是成为系统推荐下一步动作的证据。

## 已确认方向

采用“A+B 混合”的指挥台式设计：

1. A：任务队列优先。首屏左侧展示“我的下一步”，直接给用户可执行动作。
2. B：场景入口辅助。首屏右侧展示“我想做”，让用户可以主动进入参数修改、我的提交、审阅队列等流程。
3. 优先级采用“真实流程待办压过推荐”。当用户有明确流程待办时，待办排在最上面；没有待办时，工作台才基于热区和风险生成“建议从这里开始”的推荐。
4. Admin 不需要“我要治理”这类零基础入口。管理员默认知道管理语境，只展示直接管理动作、管理摘要和高优先级异常。
5. 普通用户和 Committer 是引导重点，因为他们更容易面对多个页面和入口时不知道下一步。

## 角色视角

工作台按四类视角呈现，而不是为现有六个角色分别做完全不同的页面。

### Guest

Guest 是只读观察视角。工作台应展示参数态势、风险热区和可访问入口，但不展示会导致权限拒绝的行动按钮。

Guest 的首屏重点：

- 查看参数整体态势。
- 查看热门风险区域。
- 申请或切换具备操作权限的角色。

### User

User 包括 Hardware User 和 Software User。硬件/软件差异只影响默认筛选、文案和任务来源，不拆成两套页面。

User 的首屏重点：

- 继续未提交草稿。
- 处理被退回的参数修改。
- 查看自己的提交进度。
- 当没有流程待办时，从高风险参数、项目热区或模块热区开始修改参数。

### Committer

Committer 包括 Hardware Committer 和 Software Committer。硬件/软件差异用于筛选与角色相关的审阅任务。

Committer 的首屏重点：

- 处理待审阅变更。
- 处理初始化项目审阅。
- 推进待合入或需人工确认的流程节点。
- 当没有明确待办时，创建或进入高风险专项审阅。

### Admin

Admin 不是新手引导对象。工作台应避免显示“我要治理”这种解释型入口。

Admin 的首屏重点：

- 管理后台。
- 新建项目。
- 用户管理。
- 配置脏态、待导出、权限异常、初始化项目等管理摘要。
- 原有参数态势和热区证据。

## 信息架构

页面阅读路径为：先知道今天该做什么，再选择主动入口，再查看推荐依据。

### 1. 我的下一步

首屏左侧的主区域，负责回答“现在建议先处理什么”。

展示方式：

- 1 个主行动，作为最高优先级动作。
- 2 到 4 个次行动，作为后续处理项。
- 每个行动包含标题、原因、状态、数量、来源和入口。
- 行动入口必须带上下文跳转，例如项目、模块、参数或审阅状态。

行动来源按优先级排序：

1. 明确流程待办：退回修改、草稿未提交、待审阅、待合入、初始化审阅。
2. 角色相关异常：权限异常、配置脏态、待导出、需人工确认。
3. 推荐开始动作：来自热区榜、项目风险分布、高风险参数和 AI 评分拆解。
4. 空闲状态：没有必须处理事项时，引导查看参数目录、我的提交或整体态势。

### 2. 我想做

首屏右侧的辅助入口，负责保留主动探索路径。

入口按角色和权限过滤：

- User：修改参数、我的提交、查看风险热区。
- Committer：处理审阅、查看历史提交、高风险专项。
- Admin：管理后台、新建项目、用户管理。
- Guest：查看参数、查看态势、申请权限。

不显示用户当前角色点进去会权限拒绝的入口。

### 3. 推荐依据

首屏下方保留原有看板内容，并调整标题和层级，让它变成“为什么推荐这些动作”的依据。

保留内容：

- 核心指标：参数总量、管理项目总数、修改频次、开发人员总数。
- 参数更新趋势。
- 各项目参数更新情况。
- 热门模块、项目、参数热区。
- 热榜维度切换：总榜、模块榜、项目榜、参数榜。
- AI 评分拆解：关联证据、维度得分、推荐入口。

这些内容不删除，只从“页面主任务”变为“任务推荐证据”。

## 组件设计

### `PersonalWorkbenchHero`

新的首屏指挥台容器。它承载左侧“我的下一步”和右侧“我想做”，替换当前仅展示态势 headline 的首屏结构。

职责：

- 展示角色摘要。
- 展示下一步行动列表。
- 展示按角色过滤后的场景入口。
- 保持首屏紧凑，让下方推荐依据区露出一部分。

### `NextActionList`

展示个人行动队列。

职责：

- 区分主行动和次行动。
- 展示行动原因和状态。
- 支持空状态。
- 点击行动时调用现有 `onNavigate`，并携带查询参数上下文。

### `ScenarioEntryPanel`

展示“我想做”入口。

职责：

- 根据角色和权限生成入口。
- 对 Admin 使用直接管理入口，不显示解释型治理入口。
- 对无权限入口进行过滤，而不是展示后再进入权限拒绝页。

### `DashboardEvidenceSection`

承接原有看板内容。

职责：

- 复用现有指标卡、趋势图、项目风险分布和热区排行榜。
- 维持当前时间范围选择和热榜维度选择能力。
- 将文案从“看板”调整为“推荐依据”或相近语义。

### `derivePersonalWorkbench`

新增纯函数，用于把应用状态转成工作台视图模型。

输入：

- `PrototypeState`
- 当前角色 `activeRoleId`
- 当前时间范围
- 当前热榜维度或由页面状态决定的 analytics

输出：

- `roleView`
- `nextActions`
- `scenarioEntries`
- `summary`
- `emptyState`

这个函数应独立测试，避免把复杂排序逻辑塞进 React 组件。

## 数据流

工作台数据从现有 mock state 派生，不新增后端。

1. 页面先调用 `deriveParameterHomepageAnalytics(state, timeWindow, hotspotDimension)`，获得现有看板数据。
2. `derivePersonalWorkbench` 接收 state、角色和 analytics。
3. 函数先收集真实流程待办。
4. 如果没有真实待办，再从 analytics 的 hotspots、riskBuckets 和 summary 生成推荐动作。
5. 所有行动和入口都经过权限过滤。
6. React 页面只负责渲染和导航，不在组件内部重复业务排序。

## 行动生成规则

### User 待办

User 行动优先来自：

- 与当前角色 submitter 匹配的草稿或提交轮次。
- 被退回或需要补充原因的变更请求。
- 软件用户可执行的待合入事项。
- 与当前项目相关的高风险参数推荐。

### Committer 待办

Committer 行动优先来自：

- `parameter.review` 权限可处理的待审阅请求。
- 硬件或软件角色匹配的审阅节点。
- 待处理的项目初始化审阅。
- 高风险、长时间停留或 AI 建议需要人工确认的请求。

### Admin 待办

Admin 行动优先来自：

- 配置已修改但未导出。
- 角色或用户异常。
- 项目初始化与参数库维护事项。
- 需要管理后台处理的系统性风险。

Admin 的行动文案应简洁直接，例如“打开管理后台处理 3 项待导出配置”，而不是“我要治理参数”。

### Guest 待办

Guest 不生成变更类行动。

Guest 的行动应是只读和权限引导：

- 查看参数态势。
- 查看风险热区。
- 申请权限或切换角色。

## 边界状态

### 无待办但有推荐

显示“今天没有必须处理的流程待办”，同时给出来自热区的推荐开始动作。

### 无待办也无明显风险

显示“今天没有必须处理的事项”，并提供查看参数目录、查看我的提交和查看整体态势入口。

### 坏数据引用

如果行动引用的参数、项目、提交轮次或审阅请求不存在，该行动直接丢弃。页面仍渲染其他行动和推荐依据区。

### 权限不匹配

工作台入口不应暴露当前角色无法访问的页面。必须在视图模型阶段过滤，而不是依赖路由权限页兜底。

### 移动端

移动端先展示“我的下一步”，再展示“我想做”，最后展示推荐依据。热区排行榜继续使用现有 accordion 行为。

## 保留与调整

保留：

- 时间范围选择。
- 核心指标。
- 参数更新趋势图。
- 项目风险分布图。
- 热区排行榜和 AI 评分拆解。
- 热榜维度切换。
- 热区上下文跳转。

调整：

- 顶部快捷入口收敛或移入“我想做”，避免首屏出现两套入口。
- 当前 headline 调整为角色化摘要或放入推荐依据区。
- 原“热门模块”标题可调整为“推荐依据”下的热区部分。
- 首屏视觉密度保持工作台风格，不做营销 hero。

不做：

- 不新增真实后端、数据库或 AI API。
- 不重做参数修改、参数审阅、管理后台等子页面。
- 不引入新的 UI 框架或图表库。
- 不为六个角色分别实现六套页面。

## 测试策略

### 纯函数测试

新增 `derivePersonalWorkbench` 测试，覆盖：

- User 有真实待办时，待办排在推荐前。
- User 无真实待办时，生成热区推荐动作。
- Committer 显示待审阅和高风险专项动作。
- Admin 显示直接管理动作，不显示“我要治理”。
- Guest 只显示只读和权限引导入口。
- 坏引用行动被丢弃。
- 无权限入口被过滤。

### 页面测试

更新 `ParameterManagementHomePage` 测试，覆盖：

- 页面出现“我的下一步”。
- 页面出现按角色过滤后的“我想做”入口。
- Admin 不出现“我要治理”或类似新手治理入口。
- 原核心指标仍存在。
- 原趋势图和项目风险图仍存在。
- 原热区排行榜、热榜维度和 AI 评分拆解仍存在。
- 点击行动和入口会携带正确上下文导航。

### 权限测试

更新权限相关测试，确认：

- Guest 不看到审阅和管理入口。
- User 不看到管理后台入口。
- Committer 不看到 Admin 管理入口。
- Admin 看到管理入口。

### 浏览器验证

实现后需要验证：

- 桌面首屏能同时看到“我的下一步”“我想做”和下方推荐依据露出。
- 移动端内容顺序正确，没有入口和图表挤压。
- 切换角色后首屏行动和入口随角色变化。
- 切换时间范围后推荐依据区仍能更新。
- 热区排行榜键盘操作和 accordion 行为不退化。

## 验收标准

- 用户进入 `/parameter-home` 后，首屏明确展示当前角色的下一步行动。
- 当存在真实流程待办时，待办优先于风险推荐。
- 当没有真实待办时，页面能基于热区和风险给出推荐开始动作。
- 普通用户和 Committer 能通过个人工作台知道应该修改、补充、审阅还是查看提交。
- Admin 不显示零基础“我要治理”入口，只显示直接管理动作和管理摘要。
- 原有看板功能被保留，并作为推荐依据合理下移。
- 所有入口按权限过滤，不让用户从工作台点入无权限页面。
- 页面仍符合 WiseEff 当前工作台风格，信息密度适中，避免营销化 hero。
