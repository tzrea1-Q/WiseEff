### `log_file_objects`

Stores metadata for uploaded log blobs. Local development writes bytes through the object-store adapter; the database keeps the storage key, original file name, MIME type, size, SHA-256 checksum, uploader, organization, and project.

Key relationships:

- `organization_id` references `organizations(id)`
- `uploaded_by_user_id` references `users(id)`
- `project_id` is text for M2 compatibility with existing project scoping

### `log_records`

Tracks one user-visible log upload or analysis record. Records reference the uploaded file object, status (`uploaded`, `processing`, `complete`, `failed`), archive state, current run, optional analysis question, optional related parameter, failure reason, submitter, and capture/update timestamps.

Key relationships:

- `organization_id` references `organizations(id)`
- `file_object_id` references `log_file_objects(id)`
- `current_run_id` references `log_analysis_runs(id)` via `log_records_current_run_fk`
- `submitted_by_user_id` references `users(id)`

Indexes:

- `log_records_org_project_status_idx` on `organization_id, project_id, status`
- `log_records_archive_state_idx` on `archive_state`

### `log_analysis_runs`

### `jobs`

Provides the M2 lightweight job queue surface with M3.5 leases and M5 retry/dead-letter metadata. Jobs track kind, target type/id, status, progress, current stage, error message, lease ownership, attempt count, retry visibility, dead-letter reason, and timestamps.

Key relationships:

- `organization_id` references `organizations(id)`

Indexes:

- `jobs_kind_target_idx` on `kind, target_type, target_id`
- `jobs_claimable_idx` on `kind, status, created_at, id`
- `jobs_processing_lease_idx` on `kind, lease_expires_at` where status is `processing`
- `jobs_retry_claimable_idx` on `kind, status, next_run_at, created_at, id`
- `jobs_dead_lettered_idx` on `kind, dead_lettered_at` where `dead_lettered_at is not null`

M5 retry behavior:

- `next_run_at` delays retry visibility; claimers skip queued jobs until this value is null or due.
- `dead_lettered_at` and `dead_letter_reason` preserve exhausted retry metadata while the job remains in failed status.

- `tool_call_ids`, `trace_id`, `created_at`
- `latency_ms`, `input_tokens`, `output_tokens`, `estimated_cost_usd`
- `safety_status`, `safety_reasons`, `fallback_reason`

Indexes:

- `agent_run_traces_session_idx` on `session_id, created_at desc`

### `user_password_credentials`

Stores WiseEff-owned local account credentials. Usernames are the local login identifier; local accounts do not require or return email addresses.

Key columns:

- `user_id` primary key, references `users(id)` with cascade delete
- `username` required after `0013_local_account_username.sql`
- `password_hash`
- `password_updated_at`

Indexes:

- `user_password_credentials_username_unique_idx` unique on `lower(username)`

### `auth_sessions`

Stores local account bearer sessions. The plaintext `we_local_*` token is returned only once to the caller; the database stores its SHA-256 hash.

Key columns:

- `id` primary key
- `user_id` references `users(id)` with cascade delete
- `organization_id` references `organizations(id)` with cascade delete
- `token_hash` unique
- `created_at`, `expires_at`, `revoked_at`, `last_used_at`

Indexes:

- `auth_sessions_user_id_idx` on `user_id`
- `auth_sessions_token_hash_active_idx` on `token_hash, expires_at` where `revoked_at is null`

### `local_registration_role_requests`

Stores Admin-governed role requests created when a local account self-registers for a Hardware/Software Committer role. Registration grants the matching base User role until the request is approved.

Local self-registration uses stable department organization ids after `0016_stable_local_registration_organizations.sql`: `org-hardware-department` for `硬件部` and `org-software-department` for `软件部`. The migration also moves existing local users, role bindings, sessions, registration role requests, and audit events from duplicate department-name organizations to those stable ids.

Key columns:

- `id` primary key
- `organization_id` references `organizations(id)` with cascade delete
- `user_id` references `users(id)` with cascade delete
- `current_role_id` references `roles(id)`, constrained to `hardware-user` or `software-user`
- `requested_role_id` references `roles(id)`, constrained to `hardware-committer` or `software-committer`
- `status` constrained to `pending`, `approved`, or `rejected`
- `decided_by_user_id` references `users(id)`
- `decided_at`, `created_at`

Indexes:

- `local_registration_role_requests_pending_user_role_idx` unique on `user_id, requested_role_id` where `status = 'pending'`
- `local_registration_role_requests_org_status_created_idx` on `organization_id, status, created_at desc`

## Current Boundaries

- Auth context reads users, roles, organizations, role bindings, optional local usernames, and local account sessions.
- Local account registration persists credentials, sessions, and pending committer role requests for Admin approval.
- Audit writes and queries use `audit_events`.
- M1 parameter management persists projects, modules, definitions, project values, history, drafts, review workflow records, and import batches.
- M2 log analysis persists file objects, records, runs, stages, reports, evidence, feedback, and lightweight jobs.
- M3 debugging persists devices, detected targets, debugging parameters, sessions, snapshots, node operations, and debugging events for the simulator-backed workflow.
- M4-M5 Agent persists sessions, messages, tool calls, approvals, run traces, provider latency/token/cost metadata, safety status, and fallback reasons. Production object storage, HDC gateway, and live Agent provider seams exist, while real target-environment evidence remains a pilot acceptance responsibility.
