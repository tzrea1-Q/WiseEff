### `jobs`

Provides the M2 lightweight job queue surface with M3.5 leases and M5 retry/dead-letter metadata. Jobs track kind, target type/id, status, progress, current stage, error message, lease ownership, attempt count, retry visibility, dead-letter reason, and timestamps.

Key relationships:

- `organization_id` references `organizations(id)`

Indexes:

- `jobs_kind_target_idx` on `kind, target_type, target_id`
- `jobs_claimable_idx` on `kind, status, created_at, id`
- `jobs_processing_lease_idx` on `kind, lease_expires_at` where status is `processing`
- `jobs_retry_claimable_idx` on `kind, status, next_run_at, created_at, id`
- `jobs_dead_lettered_idx` on `kind, dead_lettered_at` where `dead_lettered_at is not null`

M5 retry behavior:

- `next_run_at` delays retry visibility; claimers skip queued jobs until this value is null or due.
- `dead_lettered_at` and `dead_letter_reason` preserve exhausted retry metadata while the job remains in failed status.

## Current Boundaries

- Auth context reads users, roles, organizations, role bindings, optional local usernames, and local account sessions.
- Local account registration persists credentials, sessions, and pending committer role requests for Admin approval.
- Audit writes and queries use `audit_events`.
- M1 parameter management persists projects, modules, definitions, project values, history, drafts, review workflow records, import batches, and project parameter files with sync/writeback metadata.
- DTS P2 (migrations `0043`/`0044`) persists board-level config sets grouping `project_parameter_files` members, immutable release baselines that pin member file versions for compare/rollback/release, and a `dtc`-backed validation gate (mode `block`/`warn`/`off` via `DTS_VALIDATION_MODE`); `0044` makes `dts_config_set.project_id` cascade on project delete. Admin UI for config sets/baselines ships with P3.
- DTS P3 (migration `0045`) persists `dts_sensitive_node_rules` for path/compatible → risk-tier / `parameter:edit-critical` matching, plus structured read/search HTTP surfaces over `dts_*` and impact kinds `phandle`/`compatible`/`config-set`.
- M2 log analysis persists file objects, records, runs, stages, reports, evidence, feedback, and lightweight jobs.
- Product feedback persists Internal Beta reports, ordered attachment metadata, admin triage status, and audit-backed updates.
- M3 debugging persists devices, detected targets, debugging parameters, sessions, snapshots, node operations, and debugging events for the simulator-backed workflow.
- M4-M5 Agent persists sessions, messages, tool calls, approvals, run traces, provider latency/token/cost metadata, safety status, and fallback reasons. Production object storage, HDC gateway, and live Agent provider seams exist, while real target-environment evidence remains a pilot acceptance responsibility.
