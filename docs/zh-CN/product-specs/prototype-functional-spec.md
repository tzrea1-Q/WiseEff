# 雷泽原型功能与实现说明

> English: [English](../../product-specs/prototype-functional-spec.md)

## 1. 原型定位

当前项目是一个本地运行的前端 Web 原型，产品主题为“雷泽”，副标题为“AI 驱动的企业业务效率平台”。它通过首页和三个业务子应用展示“参数管理、日志分析、调试平台（节点调试）”等典型高频场景，并把 AI Agent 设计为跨页面的能力入口。

“雷泽”强调用 AI 辅助企业高频业务作业提效：既保留工作台所需的信息密度、流程确认和治理留痕，也通过 Agent 降低检索、分析、审阅、调试和流转成本。

原型重点不是实现真实生产系统，而是用可交互、可演示、信息密度较高的工作台界面，说明未来平台可以如何把分散工具整合到统一入口，并通过 AI 辅助提升业务处理效率。

当前原型边界如下：

- 前端单页应用：基于 Vite、React、TypeScript 实现。
- 本地状态模拟：业务数据来自 mock 数据和 React reducer，不连接真实数据库。
- AI 能力模拟：通过规则化的 Agent 计划、建议话术、动作按钮和状态变更模拟 AI 代劳流程，不调用真实大模型 API。
- 设备与日志模拟：日志上传、分析进度、样机连接、参数下发均为前端交互模拟。
- 权限与审计模拟：角色切换、权限表、审计事件用于展示治理闭环，不接入真实身份认证系统。

Implementation note: the mock prototype remains available for demos and component tests, but M0-M5 added API-mode implementations for the core parameter, log, debugging, Agent, audit, contract, and operations paths. This document still describes prototype behavior; production-oriented behavior is governed by the API/runtime docs and M5 acceptance evidence.

## 2. 页面与路由总览

| 路由 | 页面 | 展示目标 |
| --- | --- | --- |
| `/` | 首页 | 宣告“雷泽”主题，展示三大子应用入口、AI 能力区域、效率指标与治理承诺 |
| `/parameters` | 项目参数用户工作台 | 面向硬件开发人员/项目开发人员，查看、筛选、对比和提交参数修改 |
| `/parameter-review` | 参数管理员工作台 | 面向参数管理员/软件人员，审阅参数变更并推进合入上库流程 |
| `/parameter-admin` | 项目参数 Admin | 管理参数数据库、批量导入、应用指标、权限与审计 |
| `/logs` | 日志智能分析 | 上传日志、查看 AI 分析进度、阅读结果、追溯历史并与 Agent 对话 |
| `/log-admin` | 日志分析 Admin | 查看日志分析应用使用指标、记录和权限配置 |
| `/node-debugging` | 节点调试平台 | 连接调试样机、通过 HDC/ADB 读写设备节点、生成快照与操作历史（当前调试主入口） |
| `/debugging` | 参数调试平台（暂时下线） | 路由保留但不在导航展示；因设备参数重载未实现，显示不可用页并引导至节点调试 |

| `/feedback-admin` | 反馈管理 | Admin 处理 Internal Beta「问题反馈」，查看附件、备注并推进状态 |

## 3. 全局能力覆盖

### 3.1 统一应用外壳

全局外壳覆盖以下能力：

- 侧边导航：展示首页、三大应用工作台和各 Admin 页面入口。
- 当前项目上下文：通过项目选择器切换当前项目，影响参数、日志等页面的数据视角。
- 当前角色上下文：通过角色选择器切换硬件开发、项目开发、参数管理员、Admin 等演示身份。
- 页面标题与路由高亮：帮助演示时快速说明当前所在业务场景。
- 全局「问题反馈」入口：从当前页面打开 `FeedbackDialog`，带入页面路径和标题，支持图片附件。
- 统一视觉语言：所有页面共用布局、卡片、表格、状态、时间线、按钮和 Agent 悬浮入口。

### 3.2 统一数据与状态模型

原型把业务对象抽象为类型化领域模型，覆盖项目、模块、参数、变更请求、日志记录、设备、可调参数、用户、指标和审计事件。

主要数据类型包括：

- 项目与模块：用于筛选参数、日志和设备。
- 参数记录：包含当前值、推荐值、范围、单位、风险等级、模块归属等信息。
- 参数变更请求：包含当前值、目标值、提交人、状态、AI 摘要和流程位置。
- 日志记录：包含文件名、项目、设备、日志类型、分析阶段、结论、置信度、影响、证据和建议。
- 设备记录：包含样机状态、连接状态、固件、项目归属和最近连接时间。
- 可调参数：包含参数释义、范围、当前值、推荐值、调试风险和下发状态。
- 用户与权限：用于 Admin 页面展示角色、权限范围和应用治理。
- 使用指标与审计事件：用于展示平台运行数据和关键行为留痕。

### 3.3 Internal Beta 问题反馈

Internal Beta 用户可以从侧边栏「问题反馈」提交产品级反馈。弹窗要求选择反馈类型（`experience`、`data`、`export_submit`、`feature`）并填写描述，可附加多张图片作为界面或数据问题证据。提交时记录当前 `pagePath` 和 `pageTitle`，方便后续定位问题。

`/feedback-admin` 是 Admin-only 处理页面。管理员可以筛选、搜索、查看详情与附件，填写处理备注，并按 `open -> in_progress -> closed` 推进状态。该反馈闭环服务于产品内测，不等同于日志分析页面里的分析质量反馈。

## 4. 首页功能覆盖

首页用于形成第一印象，不展示“问题清单”，而是直接宣告平台能力。当前首页覆盖以下展示点：

### 4.1 平台能力宣告

首页第一屏强调“雷泽：AI 驱动的企业业务效率平台”的价值：

- 统一连接参数、日志、调试等业务效率场景。
- 通过 AI Agent 降低检索、分析、审阅和执行成本。
- 把流程推进、治理留痕和业务指标放在同一平台中呈现。
- 通过“雷泽”这一品牌主题表达闪电般的效率提升和可沉淀、可治理的业务流程承托。

### 4.2 三大子应用入口

首页提供三个核心子应用入口：

- 项目参数在线管理平台。
- 日志智能分析平台。
- 节点调试平台（`/node-debugging`）。
- 参数调试工作台（`/debugging`，2026-07-01 起暂时下线，见 `docs/exec-plans/completed/2026-07-01-wiseeff-parameter-debugging-platform-redesign.md`）。

每个入口强调业务能力和可进入的工作台，而不是停留在营销式介绍。页面通过卡片、图标、指标和交互动效提升展示性。

### 4.3 AI 能力展示

首页包含更强视觉表现的 AI 能力区域，用于突出“智能 Agent 是平台级能力，而不是单点插件”。展示内容包括：

- AI 对参数、日志、调试场景的统一理解。
- Agent 可执行的典型任务。
- 多业务域之间的数据关联感。
- 科技感更强的深色能力带，与整体浅色首页形成对比。

### 4.4 指标与治理展示

首页展示平台能够追踪的效率与治理指标，例如：

- AI 辅助处理效率。
- 参数流转效率。
- 日志分析吞吐。
- 调试闭环状态。
- 权限、审计、流程确认等治理能力。

## 5. 应用一：项目参数在线管理平台

项目参数在线管理平台覆盖用户工作台、参数管理员工作台和 Admin 管理页三个页面。

### 5.1 用户工作台

面向硬件开发人员和项目开发人员，当前覆盖以下功能：

- 查看当前库上参数信息。
- 按模块筛选参数。
- 按风险等级筛选参数。
- 按当前项目上下文查看参数。
- 查看参数名称、模块、当前值、推荐值、取值范围、单位、风险等级、最后更新时间等信息。
- 对比不同项目之间的参数差异。
- 选择参数并提交修改目标值。
- 填写修改原因。
- 通过确认弹窗提交参数修改请求。
- 查看参数修改合入上库流程。
- 查看已有变更请求状态。
- 通过悬浮 Agent 获取参数巡检、风险提示和代办动作。

用户工作台还覆盖若干演示边界状态：

- 筛选后无数据时展示空状态。
- 未选择参数时无法提交修改。
- 状态变更前需要用户确认。
- Agent 代劳动作不会绕过确认弹窗。

### 5.2 参数管理员工作台

面向软件人员/参数管理员，当前覆盖以下功能：

- 查看参数修改审阅队列。
- 查看每条变更请求的当前值、建议值、提交人、创建时间和状态。
- 查看 AI 生成的审阅摘要。
- 查看合入上库流程时间线。
- 推进参数修改请求到下一流程状态。
- 软件开发人员在「软件User合入」阶段确认合入前，必须填写有效的 http(s) 合入链接。
- 合入完成后（状态为「已合入」），审阅详情展示独立的「合入链接」卡片，并在流程时间线的「软件User合入」步骤中提供可点击外链。
- 对待审阅请求执行“进入审阅/推进合入”等演示操作。
- 使用 Agent 生成审阅建议并触发流程推进。
- 对关键流程操作进行确认。

### 5.3 项目参数 Admin 页面

面向应用管理员，当前覆盖以下功能：

- 五步批量参数导入向导：第 1 步必须选择目标项目（可新建项目）；第 2 步支持上传或粘贴 `.xlsx`、`.csv`、JSON 及 DTS 片段并展示解析报告；第 3 步逐行审阅（差异对比、编辑、跳过）；第 4–5 步为批次预览与确认应用。
- 批量参数导入入口。
- 参数数据库管理视图。
- 嵌套模块树管理（创建子模块、移动、受控删除）；父级模块筛选包含子树。
- 当前库上参数列表管理。
- 应用使用数据指标。
- 用户权限管理。
- 审计事件查看。
- 参数导入成功的前端反馈。

## 6. 应用二：日志智能分析平台

日志智能分析平台覆盖用户工作台和 Admin 管理页两个页面。

### 6.1 用户工作台

当前覆盖以下功能：

- 上传示例日志。
- 根据上传文件创建日志分析记录。
- 实时展示 AI 自动化分析进展。
- 展示解析、模式匹配、根因推断、完成等阶段。
- 展示分析进度条和阶段时间线。
- 分析完成后展示结论、置信度、影响范围、证据链和推荐动作。
- 支持根据已上传日志与 Agent 对话。
- Agent 可生成根因研判和排查清单。
- 查看历史日志分析记录。
- 对不支持格式展示错误提示。

日志智能分析的演示重点是“过程可见”。用户不仅能看到最终结论，也能看到 AI 正在执行哪些阶段，这适合向专家展示系统可信度和可解释性。

### 6.2 日志分析 Admin 页面

当前覆盖以下功能：

- 查看日志分析应用使用指标。
- 查看日志分析记录概览。
- 查看分析状态、置信度、影响范围等管理视角信息。
- 管理日志分析应用后台人员权限。

## 7. 应用三：调试平台

调试平台当前以 **节点调试** 为主入口；**参数调试** 工作台暂时下线。

### 7.1 节点调试用户工作台（`/node-debugging`，当前主路径）

当前覆盖以下功能：

- 本地 Device Bridge 安装、配对与连接引导。
- 查看调试样机列表和连接状态。
- 通过 HDC/ADB 读写设备节点值。
- 读回校验、操作历史与快照记录。
- 高风险写入确认与审计留痕。

### 7.2 参数调试用户工作台（`/debugging`，暂时下线）

2026-07-01 起，该路由不在主导航展示，直接访问会显示「页面暂时不可用」，并引导用户进入节点调试。原因：尚未实现真实的 **设备参数重载** 底层能力；若继续展示「连接 → 改目标值 → 下发 → 回滚」流程会误导用户。

原型的 `DebuggingPage` 组件仍保留在代码库中，供后续参数重载能力就绪后恢复。恢复前需完成参数库 1:1 对齐、参数调试与节点调试管理分离，以及 reload 运行时（见 **TD-032**）。

历史上该页面曾覆盖：

- 查看调试样机列表和连接状态。
- 查看所有支持实时调节的参数及目标值编辑。
- 在连接样机后执行参数调节与回滚准备展示。

### 7.3 参数调试 Admin 页面

当前覆盖以下功能：

- 查看调试平台使用指标。
- 查看设备连接与在线状态。
- 查看支持实时调节的参数目录。
- 管理调试平台后台人员权限。

## 8. AI Agent 功能覆盖

AI Agent 是当前原型最重要的横向能力之一。它以悬浮图标形式存在，可以展开或缩起，在不同页面提供与当前业务上下文相关的建议和动作。

### 8.1 Agent 交互形态

当前覆盖以下能力：

- 页面右下角悬浮入口。
- 展开后显示 Agent 面板。
- 展示当前上下文摘要。
- 展示建议提问。
- 展示 AI 思考步骤。
- 展示可执行动作按钮。
- 对部分动作提供执行反馈。
- 对涉及状态变更的动作弹出确认框。
- 不同页面展示不同 Agent 能力。

### 8.2 Agent 场景覆盖

在项目参数用户工作台，Agent 可以：

- 帮助筛查高风险参数。
- 辅助生成参数修改建议。
- 发起参数修改提交流程。
- 解释项目间参数差异。

在参数管理员工作台，Agent 可以：

- 总结待审阅请求。
- 给出 AI 审阅摘要。
- 辅助推进审阅流程。
- 提醒高风险参数需要人工确认。

在日志智能分析页面，Agent 可以：

- 解读当前日志分析结果。
- 输出根因判断。
- 生成排查清单。
- 关联历史记录进行说明。

在节点调试页面，Agent 可以：

- 根据当前设备与节点读写上下文给出建议。
- 提醒连接状态、协议与风险等级。

参数调试页面（`/debugging`）Agent 能力随页面暂时下线一并不可用；恢复参数重载后重新启用（**TD-032**）。

在首页，Agent 更偏向平台级能力介绍，用于说明 AI 可以跨应用理解业务上下文。

## 9. 视觉与交互实现策略

当前原型的视觉目标是：首页具有更强展示性，工作台更强调信息清晰，AI 相关能力要更有记忆点。

### 9.1 首页视觉策略

首页偏浅色系，使用更开阔的布局和更强的首屏展示效果：

- 首屏直接宣告能力，不把问题作为主视觉内容。
- 使用清晰的业务入口卡片承接三大子应用。
- 使用指标和能力模块形成“平台已经具备能力”的观感。
- AI 能力区域使用更强对比和动态感，使其成为视觉记忆点。

### 9.2 工作台视觉策略

工作台更偏企业级生产力工具：

- 信息层级清晰。
- 表格可扫读。
- 筛选条件直接可见。
- 状态、风险和阶段用统一徽标表达。
- 动效适度，不干扰密集信息阅读。
- 操作按钮靠近业务对象，降低演示时的解释成本。

## 10. 当前限制

当前原型为了保证演示效率，刻意将复杂后端能力前端化、可视化、可控化。主要限制包括：

- 没有真实登录、组织、租户和 RBAC 权限系统。
- 没有真实数据库，刷新页面后会回到初始 mock 状态。
- 没有真实参数库、版本库或上库流水线。
- 没有真实日志文件解析，只模拟上传和分析阶段。
- 没有真实大模型调用，Agent 输出由前端规则生成。
- 没有真实设备连接、通信协议或参数下发通道。
- 没有生产级审计存储，审计事件只用于前端展示。
- 没有真实文件批量导入解析。
- 没有后端 API、错误重试、并发控制和权限校验。

这些限制是原型阶段的合理取舍。它们让项目可以快速形成可演示体验，同时保留清晰的后续工程落点。

M0-M5 have addressed many of these limitations in API mode. Remaining production gaps should be tracked through `docs/generated/m5-pilot-acceptance.md` and `docs/exec-plans/tech-debt-tracker.md`, not by editing prototype behavior claims into product signoff.

## 11. 后续工程演进建议

如果从原型进入真实产品建设，建议按以下方向演进：

### 11.1 后端与数据层

- 建立参数库、日志记录、设备记录、用户权限和审计事件的数据模型。
- 提供 REST 或 GraphQL API。
- 将当前 `mockData.ts` 替换为 API 查询。
- 将 reducer action 映射到真实后端 mutation。
- 引入持久化、分页、搜索、排序和权限过滤。

### 11.2 AI 能力层

- 接入真实大模型或企业内部模型服务。
- 将 `createAgentPlan()` 升级为“上下文组装 + 工具调用 + 人类确认”的 Agent 编排层。
- 为参数审阅、日志分析、调试推荐分别建立工具接口。
- 增加 AI 输出引用来源、证据链、置信度解释和人工反馈闭环。
- 对所有会改变生产状态的 AI 动作保留确认、审批和审计。

### 11.3 参数管理工程化

- 对接真实参数数据库和版本管理系统。
- 建立参数变更审批流。
- 支持参数 diff、回滚、合入记录和发布记录。
- 支持批量导入模板校验、冲突检测和影响分析。

### 11.4 日志分析工程化

- 支持真实日志上传、解析、索引和存储。
- 增加日志类型识别、异常模式匹配、相似案例召回。
- 支持长任务队列和后端分析进度推送。
