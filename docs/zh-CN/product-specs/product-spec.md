# WiseEff 产品 Spec

> English: [English](../../product-specs/product-spec.md)

日期：2026-05-25
版本：v0.1 产品化基线

## 1. 背景

WiseEff 是一个面向企业高频工程作业的 AI 协同效率平台。当前项目已经实现了可交互前端原型，覆盖参数管理、日志分析、参数调试、统一 Agent、权限演示和审计展示。原型价值已经验证在交互和信息架构层面，下一阶段目标是把它发展为可真实使用、可测试、可上线的企业级系统。

正式开发阶段不应重写现有原型，而应把当前 `domain`、`application/ports`、`infrastructure/mock` 和 DTO 边界作为迁移起点。第一阶段以参数管理闭环为主，逐步引入后端 API、数据库、认证授权、审计、异步任务、真实日志分析、设备网关和 Agent 编排。

Implementation note: M0-M5 have already moved the core workflows behind governed API/runtime seams. The next product decision point is not broad new scope; it is staging/pilot evidence collection and focused production hardening while preserving the user-visible workflow intent below.

## 2. 产品定位

WiseEff 的定位是“业务流程里的 AI 协同工作系统”，不是单纯看板、聊天机器人或内部工具集合。它要把参数、日志、设备调试和审批治理放在一个统一工作台中，让用户可以：

- 看到自己当前应该处理的工程任务。
- 在可审计的流程中提交、审阅、合入和回滚参数变更。
- 上传日志并获得可解释的分析结果和证据链。
- 连接调试设备，安全读取和写入允许范围内的参数。
- 让 AI Agent 辅助检索、总结、建议和准备动作，但不绕过权限、审批和人工确认。

## 3. 用户与角色

WiseEff 的正式角色以现有前端权限模型为基础，后续在后端持久化：

| 角色 | 主要诉求 | 核心权限 |
| --- | --- | --- |
| Guest | 只读查看平台能力和公开数据 | 查看可公开页面和只读参数 |
| Hardware User | 提交硬件相关参数修改、上传日志、使用调试能力 | 参数查看、参数草稿、日志上传、设备调试 |
| Software User | 提交软件/配置相关修改、跟踪合入状态 | 参数查看、参数草稿、日志上传 |
| Hardware Committer | 审阅硬件侧参数变更 | 参数审阅、退回、推进硬件流程节点 |
| Software Committer | 审阅软件侧参数变更并推进合入 | 参数审阅、推进合入、处理退回 |
| Admin | 管理项目、参数库、用户、权限、审计和系统配置 | 管理后台、用户管理、审计查询、配置管理 |

## 4. 产品目标

MVP 成功标准：

- 参数管理可以脱离 mock 数据，在真实数据库中完成项目、参数、草稿、提交、审阅、合入、历史和审计闭环。
- 日志分析可以接收真实文件，创建分析任务，展示进度和结果，并将证据关联回参数。
- 调试平台可以通过受控网关读取和写入设备节点，并生成调试会话记录。
- Agent 可以基于当前页面上下文启动会话，生成建议和工具调用申请；所有变更型工具调用必须等待批准。
- 系统具备基础企业级能力：登录、RBAC、审计、错误处理、测试、CI、部署、监控和备份。

非目标：

- MVP 不追求一次性覆盖所有企业集成系统。
- MVP 不允许 Agent 自动执行生产变更。
- MVP 不把设备网关和主业务后端强耦合在同一进程内。
- MVP 不保留前端 mock 作为生产数据源。

## 5. 核心业务域

### 5.1 参数管理

参数管理是第一优先级业务域。它负责参数库治理、项目参数视图、参数变更提交、审阅、合入、历史追踪和批量导入。

核心能力：

- 项目参数列表、筛选、搜索、排序和详情查看。
- 参数定义与项目参数值分离。
- 参数草稿支持多条变更批量提交。
- 变更请求进入固定审批流：提交、硬件检视、软件检视、软件合入、已合入或已打回。
- 高风险参数必须明确展示风险、影响范围、原因和审阅结论。
- 参数历史保留版本、变更人、变更时间和关联请求。
- 批量导入必须校验格式、冲突、范围和权限。
- 所有写操作产生审计事件。

### 5.2 日志分析

日志分析负责真实日志上传、解析、分析任务、证据链、结果查看和历史追溯。

核心能力：

- 支持 `.log`、`.txt`、`.json` 等文本日志。
- 文件上传后创建 `LogRecord` 和 `LogAnalysisRun`。
- 分析过程分为解析、模式匹配、根因推断、报告生成。
- 前端展示阶段进度、结论、置信度、证据行号、建议动作和失败原因。
- 分析结果可以关联参数、设备和后续处理动作。
- 分析任务必须可重试、可失败、可取消或归档。

### 5.3 参数调试

参数调试负责设备连接、节点读取、参数写入、回读校验、回滚准备和调试审计。

核心能力：

- 通过设备网关检测可用目标。
- 读取设备节点并展示当前值、访问模式和状态。
- 对可写节点执行写入和可选回读校验。
- 高风险写入必须要求确认和权限校验。
- 调试会话记录设备、用户、节点、写入值、回读结果、耗时和错误。
- 回滚必须基于写入前快照，不允许无快照回滚。

### 5.4 Agent 协同

Agent 是跨业务域的辅助层。它不拥有业务状态，只通过受控工具调用访问系统能力。

核心能力：

- 根据页面、项目、角色和对象上下文启动会话。
- 生成解释、总结、推荐和下一步建议。
- 对读操作可自动执行，但仍需遵守权限边界。
- 对写操作生成审批申请，用户批准后才执行。
- 每次工具调用必须记录输入、输出、执行人、审批人和审计关联。
- Agent 输出要标记依据来源、置信度和可执行动作边界。

## 6. 关键工作流

### 6.1 参数变更闭环

1. 用户进入参数工作台。
2. 系统按项目、模块、风险和权限返回参数。
3. 用户选择参数并创建修改草稿。
4. 系统校验目标值、范围、重复提交和权限。
5. 用户提交一轮参数变更。
6. 审阅人按流程节点处理请求。
7. 合入成功后生成新参数历史。
8. 系统写入审计事件并通知相关用户。

### 6.2 日志分析闭环

1. 用户上传日志并填写可选分析问题。
2. 后端保存文件并创建分析任务。
3. 任务队列执行解析、模式匹配、根因推断和报告生成。
4. 前端通过轮询或服务端事件查看进度。
5. 用户查看证据链和建议动作。
6. 用户可跳转到相关参数或创建后续处理任务。

### 6.3 设备调试闭环

1. 用户进入调试平台并检测设备。
2. 系统展示可读写节点和访问模式。
3. 用户读取当前值并选择目标值。
4. 系统做权限、范围、风险和设备状态校验。
5. 用户确认写入。
6. 设备网关执行写入和回读校验。
7. 系统生成调试事件、快照和审计记录。

### 6.4 Agent 工具调用闭环

1. 用户打开 Agent。
2. 系统创建会话并注入页面上下文。
3. Agent 返回建议、解释和工具调用。
4. 读操作在权限允许时执行。
5. 写操作生成审批卡片。
6. 用户批准后，后端执行工具并写入审计。
7. Agent 汇总结果并展示下一步。

## 7. 非功能要求

| 类别 | 要求 |
| --- | --- |
| 可用性 | 普通页面 P95 API 响应低于 800ms；长任务通过进度事件反馈 |
| 安全 | 企业 SSO/OIDC、后端 RBAC、审计不可篡改、敏感日志脱敏 |
| 可靠性 | 后端写操作具备幂等键和乐观锁；任务队列可重试 |
| 可测试 | 所有领域规则有单元测试；API 有集成测试；核心流程有 E2E |
| 可观测 | API、任务、Agent、设备网关都有日志、指标和 trace id |
| 可维护 | 前后端共享 API 合同；领域逻辑不写在页面组件里 |
| 可上线 | 环境隔离、数据库迁移、回滚策略、健康检查和备份齐备 |

## 8. 验收标准

MVP 完成时必须满足：

- `npm test` 和 `npm run build` 通过。
- 后端单元测试、集成测试和数据库迁移测试通过。
- 参数变更闭环 E2E 通过：登录、查询、草稿、提交、审阅、合入、历史、审计。
- 日志分析 E2E 通过：上传、进度、完成、证据、失败重试。
- 设备网关在模拟器环境通过读写和回滚测试。
- Agent 写操作无法绕过审批。
- 审计记录可按用户、项目、业务对象和时间查询。
- 生产配置中 mock runtime 默认关闭。
