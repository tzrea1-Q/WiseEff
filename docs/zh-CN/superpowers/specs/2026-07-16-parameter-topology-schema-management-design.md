# 面向拓扑与 Schema 的参数精细化管理——设计规格

> English: [English](../../../superpowers/specs/2026-07-16-parameter-topology-schema-management-design.md)

**日期：** 2026-07-16

**状态：** Brainstorming 设计已确认，等待书面规格审阅

**范围：** 参数库、项目参数、DTS 配置集、Schema 治理、历史迁移和编译验证

**首期交付：** DTS 完整闭环；JSON 和人工参数只迁移统一身份模型

## 1. 摘要

WiseEff 不能再把 DTS 属性的完整路径当成参数名。以下路径：

```text
amba.i2c@FDF5E000.sc8562@6E.gpio_int
```

实际包含多个不同概念：

- `gpio_int` 是属性键。
- `sc8562` 是驱动或器件模块。
- `sc8562@6E` 是具体项目中的器件实例。
- `i2c@FDF5E000` 和 `amba` 是拓扑祖先。
- `/amba/i2c@FDF5E000/sc8562@6E` 是某个有效树版本中的定位路径。
- `<&gpio13 29 0>` 是项目实际值。
- “充电 → 电源 IC → Charge Pump”是业务治理分类。

这些概念必须拥有独立身份和明确关系。本规格采用四层模型：

1. 不可变源码 occurrence 与解析后的有效拓扑。
2. 版本化的驱动规格和属性规格。
3. 项目实例绑定及实际生效值。
4. 与技术模型正交的业务治理。

完整 DTS 配置集，而不是单个文件，是语义解析和编译的最小单元。生产发布必须通过确定性身份匹配、Schema 校验、引用完整性校验和真实 `dtc` 工具链，并采用失败关闭策略。

## 2. 现状证据与问题

对用户提供的板级 overlay 和当前种子实现进行全量审计后得到：

- 50 个节点和 170 个属性。
- 169 个属性定义错误地把完整路径作为显示名称。
- 24 种重复属性键，其中 `status` 38 次、`compatible` 5 次、`reg` 4 次、`gpio_int` 2 次。
- 84 个字符串列表、61 个 u32 数组、11 个混合值、6 个 phandle 列表、5 个 byte 数组、2 个布尔属性和 1 个空值。
- 28 个外部 overlay target 在当前文件中没有 `compatible`。
- 18 条 phandle 引用，其中 12 条依赖 base tree。
- 14 条候选字符串表引用，其中 9 条无法在单文件中解析。

现有结构解析器已经保留节点名和单元地址。数据语义在后续种子派生时丢失：完整属性路径被写入 `name`，`module` 则通过路径关键词推断。前端合同仍是扁平的 `name/module/sourceNodePath`，无法分别表达源码 occurrence、合并拓扑、驱动身份和 Schema 身份。

只修正界面标签会掩盖问题，但路径身份、overlay、历史追溯和安全回写仍不可靠。

## 3. 已确认的设计决策

| 主题 | 决策 |
| --- | --- |
| 目标 | 设计完整目标架构，并提供分阶段交付路线 |
| 统一模型 | 使用一个 `ParameterSpec`，DTS、JSON 和人工参数作为来源子类型 |
| DTS 语义单元 | 完整配置集，包括入口 DTS、DTSI/include、base tree、overlay、顺序、include 路径和编译配置 |
| Schema 来源 | Linux dt-schema 优先，组织级 Vendor Schema 扩展其次，审核后的人工规格最后 |
| 未知属性 | 生成带证据的推理草稿；审核前不能用于发布 |
| 技术模块与业务模块 | 驱动/器件模块和业务分类是两个正交维度 |
| 身份歧义 | 只接受确定性匹配；歧义项进入人工映射队列并阻止发布 |
| 治理范围 | 平台标准 + 组织级 Vendor 扩展；项目只绑定和值配置，不能修改共享规格 |
| “推荐值” | 改名为“示例值”；Schema 默认值、策略目标值和项目生效值分别建模 |
| 首期格式范围 | DTS 实现导入、管理、编辑、导出和编译闭环；JSON/人工参数不新增专用解析器 |
| 历史迁移 | 一次性替换旧 ID；所有草稿、变更、审批、审计和基线全部重映射 |
| 切换方式 | 维护窗口停写，全量校验后原子切换；失败时整体回滚 |
| 发布门禁 | 生产发布必须通过确定性映射、Schema、引用完整性和真实工具链 |

## 4. 候选方案与结论

### 4.1 修补现有扁平模型

把路径最后一段拆成显示名并调整前端。成本低，但仍使用路径身份，无法安全处理 overlay、include 图、重复属性、地址变化、Schema 版本和历史重映射。

### 4.2 以有效路径为中心

建立有效节点树，并用完整路径识别参数。它改善了层级展示和合并，但节点移动、单元地址变化或 overlay 重组仍会导致身份漂移，源码无损回写和跨项目规格复用也难以保证。

### 4.3 四层语义模型

分别存储源码 occurrence、有效逻辑节点、版本化规格、项目绑定和业务治理。迁移成本最高，但只有该方案能同时满足稳定身份、完整 DTS 语义、精细治理、历史审计和编译验证。

**结论：**采用四层语义模型。

## 5. 目标与首期非目标

### 5.1 目标

- 支持完整 DTS 工程，包括 `/include/`、base DTS/DTSI、外部 target、label、phandle 和有序 overlay。
- 区分属性规格、驱动类型、器件实例、拓扑位置、源码声明和业务分类。
- 保留原始源码用于审计和最小化回写，同时提供解析后的有效树。
- 将已知属性绑定到版本化的 Linux 或 Vendor Schema。
- 为每个项目值记录明确来源、覆盖链、校验状态和审计历史。
- 集成可复现的 `dtc`、`fdtoverlay` 和 dt-schema 部署与发布门禁。
- 把全部现有参数及历史数据迁移到新身份模型，不长期维护兼容投影视图。

### 5.2 首期非目标

- 为 JSON 和人工参数新增专用解析器或编辑器。
- 自动执行 Git commit、PR 或仓库 webhook。
- 自动批准推理生成的属性规格。
- 静默修复非法值或无法解析的引用。
- 部分生产切换或永久双写。

## 6. 核心架构

### 6.1 源码与有效拓扑层

`DtsConfigSet` 是最小可构建、可发布单元，记录：

- 项目和板级变体；
- 入口文件和成员角色；
- include 搜索路径；
- base tree 关系；
- overlay 应用顺序；
- 可选的确定性预处理配置；
- 编译参数和工具链配置；
- 不可变配置版本和发布基线。

每次导入生成不可变 `SourceFileVersion`。无损解析器生成：

- `NodeOccurrence`：源码中的节点或 target 片段；
- `PropertyOccurrence`：属性声明、布尔属性、覆盖、删除或引用；
- 源文件、行列范围、token 范围、原始值、解析 AST 和内容哈希。

有效树解析器应用 include 和 overlay，生成 `LogicalNodeInstance` 和有效属性值。每个有效节点和属性都保留完整来源链，说明哪些 occurrence 创建、替换或删除了它。

源码 occurrence ID 只在单个源码版本中有效。逻辑节点 ID 在有确定性证据证明连续性时跨配置版本保持稳定。

### 6.2 驱动与属性规格层

`ParameterSpec` 是统一规格根实体，不同来源提供专用子类型：

- DTS：`DriverSchema` 和 `PropertySpec`。
- JSON：key/path/value-shape 元数据。
- 人工参数：明确治理的类型和约束。

`DriverSchema` 表示版本化的驱动或器件合同，记录 `compatible` 模式、父节点/总线约束、来源、归属和生命周期。

`PropertySpec` 由 Schema 命名空间、Schema 版本和属性键共同识别，记录值结构、单位、约束、引用、Schema 默认值、示例值、文档和废弃状态。属性键本身不是全局身份：`SC8562/gpio_int` 和 `MT5788/gpio_int` 是不同规格。

规格优先级为：

1. 固定版本的 Linux dt-schema。
2. 固定版本的组织级 Vendor Schema。
3. 已审核人工规格。
4. 未审核推理草稿；不可发布。

组织扩展可以增加 Vendor 属性或收紧组织策略，但不能静默改写已固定的平台 Schema 版本。Schema 升级产生新的不可变版本，必须显式执行影响分析。

### 6.3 项目实例与值绑定层

`ProjectPropertyBinding` 连接：

- 项目和配置版本；
- 一个 `LogicalNodeInstance`；
- 一个确定版本的 `PropertySpec`；
- 类型化的实际生效值；
- 来源和覆盖链；
- Schema、策略和编译状态；
- 相对基线差异和历史。

完整节点路径只是某个版本中的定位器，不是主键。`sc8562@6E`、I²C 地址、GPIO 引用、状态和板级值等项目事实属于实例或绑定，不属于共享规格。

原“推荐值”字段被移除：

- `exampleValue` 只说明语法，不参与校验。
- `schemaDefault` 来自固定版本 Schema。
- `policyTarget` 来自组织或产品策略，可参与合规判断。
- `effectiveValue` 是项目配置版本中的解析后生效值。

### 6.4 业务治理层

`BusinessCategory` 是独立层级。驱动规格和逻辑实例都可以关联多个业务分类。治理元数据包括：

- 责任团队和负责人；
- 风险与敏感等级；
- 审批策略；
- 权限；
- 适用的校验策略；
- 测试套件。

“SC8562 驱动”和“充电 → 电源 IC → Charge Pump”因此相互独立，但都能用于搜索和治理。

## 7. 身份解析

### 7.1 规格身份

Schema 匹配按以下证据执行：

1. 有效 `compatible`；
2. 父总线和 Schema 约束；
3. 节点结构及必需/允许属性；
4. 固定的 Schema 命名空间与版本；
5. 已审核的组织级显式映射。

仅有一个有效结果时建立规格绑定；没有结果时创建推理规格草稿；存在多个结果时创建阻断式映射任务。

外部 overlay target 不能获得伪造的驱动身份。解析器必须等待 base tree；缺少 base 的配置集标记为不完整且不可发布。

### 7.2 逻辑实例连续性

跨版本身份引擎只接受确定性证据：

- 已审核的显式连续性映射；
- 稳定的父逻辑节点身份；
- 唯一驱动/Schema 身份；
- `reg` 或 Schema 声明的其他唯一键；
- 单元地址和拓扑关系；
- Schema 明确声明具有身份意义的稳定标识。

路径、label、显示名或模糊评分不能单独证明连续性。存在多个候选时创建 `IdentityMappingTask`，授权审核者确认前阻止发布。

每次人工决策保存候选项、证据、审核者、原因、时间和受影响版本。

## 8. DTS 值模型

DTS 值不能压缩成无类型字符串。`DtsValue` 必须覆盖：

- 以存在表示的布尔属性；
- 字符串和字符串列表；
- 默认或显式 `/bits/` 宽度的 cell 数组；
- byte 数组；
- 带参数 cell 的 phandle 引用；
- 多段和混合表达式；
- 对命名参数表的字符串引用；
- 空值；
- 属性/节点删除和 overlay 动作。

每个属性 occurrence 同时保存：

1. **原始表示：**精确 token 和 trivia，用于审计与未改区域字节保持。
2. **类型化 AST：**用于编辑、类型校验、引用解析和渲染。
3. **规范化表示：**用于语义比较、索引、策略判断和稳定 diff。

回写时未修改区域必须保持字节一致；修改值尽量沿用局部格式。修改后的文本即使不能全文保持一致，重新解析后的有效树也必须语义等价。

## 9. 组件边界

| 组件 | 职责 | 主要依赖 |
| --- | --- | --- |
| 配置集仓库 | 文件、不可变版本、成员角色、构建配置和基线 | 数据库、对象存储 |
| 无损 DTS 解析器 | 源码 occurrence、token 范围和类型化 AST | 源码字节 |
| 有效树解析器 | include、label、phandle、overlay/delete 语义和来源链 | 解析器、配置清单 |
| Schema 注册中心 | Linux Schema、Vendor 扩展、版本和审核状态 | Schema 存储 |
| Schema 匹配器 | 驱动和属性候选解析 | 有效树、注册中心 |
| 身份引擎 | 跨版本逻辑实例连续性和映射任务 | 有效树、已审核映射 |
| 参数管理服务 | 规格、绑定、分类、编辑、审批和搜索 | 各领域仓库 |
| 校验与编译服务 | 引用、Schema、策略、`dtc` 和 overlay 编译 | 工具链运行器 |
| 审计与发布服务 | 不可变事件、基线、原子发布和回滚证据 | 全部领域服务 |

每个组件输出类型化结果和结构化诊断。解析器不审核 Schema，前端不推断身份，编译器不直接修改领域状态。

## 10. 端到端数据流

### 10.1 导入与解析

1. 将配置清单和全部源码冻结为新的不可变版本。
2. 无损解析全部文件，生成节点和属性 occurrence。
3. 解析 include、label、phandle、base target、overlay 和删除语义。
4. 构造有效逻辑树及完整来源链。
5. 匹配驱动 Schema 和属性规格。
6. 确认逻辑身份，或创建阻断式映射任务。
7. 创建项目绑定和类型化实际值。
8. 执行引用、Schema、组织策略和内部一致性校验。
9. 使用真实工具链独立编译固定配置。
10. 对比内部解析结果与编译器结果，保存诊断和产物哈希。

内部解析器负责源码映射和编辑；`dtc`/`fdtoverlay` 是编译事实来源。两者交叉检查，避免内部实现自证正确。

### 10.2 编辑与发布

1. 用户编辑类型化项目绑定，而不是拼接 DTS 文本。
2. 服务校验值结构，并显示目标文件、occurrence、overlay 和影响范围。
3. 默认不修改共享 base；项目差异更新或新建明确 overlay。
4. 最小源码 patch 产生新的不可变文件和配置版本。
5. 重新执行完整解析和校验流水线。
6. 成功版本进入现有审批流程。
7. 所有阻断门禁通过后，发布操作原子推进发布基线。

任何失败都不能覆盖上一个已发布版本。

## 11. 产品信息架构

### 11.1 参数库

参数库管理“规格”，主要列和筛选项为：

- 属性键；
- 驱动/器件模块；
- `compatible`；
- 值类型、单位和约束；
- Schema 来源及版本；
- 示例值；
- 业务分类；
- 审核/废弃状态；
- 项目和实例使用数量。

未知或推理属性进入专门审核队列，展示证据、候选 Schema、冲突和受影响项目。

### 11.2 项目参数

项目页面管理“实例”，分为：

- **设备树：**展示 `amba → i2c@FDF5E000 → sc8562@6E` 等真实层级，以及总线、驱动、地址、状态和校验标记。
- **属性表：**展示属性键、类型化实际值、来源、Schema 状态、策略状态和基线差异。
- **详情面板：**展示规格、项目值、源码 occurrence、覆盖链、业务治理、诊断和历史。

页面可切换：

- **有效树：**最终解析配置，用于管理和校验。
- **源码树：**全部 DTS/DTSI/overlay occurrence 及其行号、覆盖或删除关系。

重复的 `&amba` 片段在源码树中保持独立，在有效树中合并为同一个逻辑节点。缺失 base target 时显示“配置集不完整”，不能猜测层级。

搜索支持属性键、驱动、`compatible`、地址、当前路径、业务分类、源码文件和原始/规范化值。搜索 `gpio_int` 会返回 SC8562 和 MT5788 的不同绑定。

## 12. 校验状态与诊断

配置版本遵循：

```text
草稿 → 解析中 → 待映射 | 无效
     → 已解析 → 已校验 → 已编译 → 待审批 → 已发布
```

以下情况阻止审批或发布：

- 语法错误或缺少 include/base 成员；
- 无法解析的 overlay target、phandle 或必需表引用；
- 驱动、属性或逻辑实例身份歧义；
- 使用未审核推理规格；
- 类型、Schema 或阻断级组织策略失败；
- `dtc`、`fdtoverlay` 或 dt-schema 不可用；
- 编译错误或策略配置为阻断的 warning；
- 内部有效树与工具链结果不一致。

诊断使用稳定错误码，包含阶段、源码文件、行列、逻辑节点、属性、严重级别、原始信息和可执行修复建议。warning 不能被静默丢弃。

并发编辑使用乐观版本检查。过期编辑返回结构化冲突并要求重新比较，禁止后写覆盖先写。

## 13. 工具链与部署

保留现有仓库命令作为统一入口：

- `npm run dtc:bootstrap`
- `npm run dtc:check -- --required`
- `npm run dtc:seed:compile`

自托管 Linux runtime 镜像必须包含固定兼容版本的 `dtc`、`fdtoverlay` 和 dt-schema。镜像构建和启动健康检查都要报告版本。每次校验记录：

- runtime 镜像/构建身份；
- 编译器和 Schema 工具版本；
- 编译配置及参数；
- 输入清单和内容哈希；
- 诊断与产物哈希。

生产发布始终采用失败关闭的 `block` 行为。现有 `warn` 和 `off` 只保留为本地诊断手段，不能生成正式发布基线。

运行器使用独立临时目录、固定参数构造、最小环境变量、无网络假设、输入大小和并发限制、硬超时及可靠清理。源码按不可信输入处理，子进程不能继承密钥。

如果项目需要确定性预处理，则把预处理程序、include 根目录、宏定义和版本纳入配置集编译配置，并像 `dtc` 一样固定和审计。

## 14. 迁移与切换

实现分阶段推进，但生产数据只切换一次：

1. 建立 Schema 注册中心、值模型、解析器和强制工具链门禁。
2. 建立源码/有效拓扑、稳定身份和映射队列。
3. 建立新 API 和产品页面。
4. 在隔离环境反复演练全量迁移，解决全部歧义和编译失败。
5. 进入维护窗口，停写、迁移、校验并原子切换。
6. 观察新系统；旧表只作为受限审计归档，不再提供业务投影。

迁移按 Schema 身份去重，而不是按源码 occurrence 去重：

- 不同项目的 SC8562 `gpio_int` 复用一个 `PropertySpec` 版本；
- SC8562 和 MT5788 的 `gpio_int` 不共享规格；
- 地址、路径和实际值保留为项目实例数据；
- `status` 等公共属性只有在 Schema 明确表示同一规格时才能共享。

全部项目值、草稿、变更单、审批、审计事件和基线都重映射。`LegacyMigrationEvidence` 保存旧 ID、旧完整名称、原定位路径、原值、源码记录哈希、新 ID 和迁移批次。

维护窗口步骤：

1. 停止参数写入，生成数据库和对象存储一致性快照。
2. 执行最终确定性迁移。
3. 校验计数、外键、映射覆盖率、历史链、源码哈希、有效树、Schema 和全部配置集编译。
4. 运行黄金样例验收。
5. 在一个发布动作中切换数据库 Schema、API 和前端。
6. 任一环节失败时整体恢复快照和旧应用。

不能向用户开放部分迁移结果。迁移演练和客户端清单证明没有遗留消费者后，旧身份 API 在切换时下线。

## 15. 测试与验收

### 15.1 单元与合同测试

- 覆盖全部支持的 DTS 值和 overlay 构造。
- include、base tree、label、phandle、覆盖和删除解析。
- Schema 优先级及不可变版本。
- 确定性身份、歧义候选和人工审核映射。
- 原始值/AST/规范化值转换与最小 patch。
- 权限、审批、审计、乐观锁和诊断合同。

### 15.2 黄金样例

用户提供的 DTS 作为永久黄金语料，验收要求：

- 50 个节点和 170 个属性全部保留；
- 所有属性显示名都是属性键，不再是完整路径；
- SC8562 和 MT5788 的 `gpio_int` 绑定不同规格；
- 重复 `&amba` occurrence 正确合并；
- 18 条 phandle、重复属性和表引用全部解析，或生成明确阻断诊断；
- 三个种子项目包含有意设计且符合 Schema 的差异。

### 15.3 端到端与真实工具

必须验证：

```text
导入 → 解析 → Schema 绑定 → 编辑 → 最小回写
     → 重新解析 → dt-schema/dtc 编译 → 审批 → 发布 → 导出
```

未修改源码保持字节一致；修改后的输出必须重新解析为语义等价的有效树，并通过真实 Linux 工具链。CI 使用与部署一致的工具链，不能只运行测试替身。

迁移演练要求旧记录 100% 映射、历史与审批链完整、同一快照迁移结果确定一致、故障注入后无部分状态，并能恢复旧系统。

涉及前端的实现必须使用真实 API，在 1440×900、768×1024 和 390×844 三种视口执行浏览器验收，覆盖参数库、源码/有效树切换、搜索、推理规格审核、身份映射、类型化编辑、编译错误、审批和发布阻断，并保留 snapshot、screenshot、控制台和相关网络检查证据。

## 16. 可观测性与安全

运行指标包括：

- 解析、Schema 校验和编译耗时；
- 各阶段失败率和诊断错误码；
- 待处理身份映射数量；
- 推理规格积压；
- 工具链健康和版本漂移；
- 发布成功率和回滚次数。

日志只保存必要诊断片段和哈希，不记录完整敏感源码。规格审核、身份映射、编辑、校验、基线、发布和迁移操作全部产生不可变审计事件。组织 Schema 覆盖、项目编辑、映射审核和发布都必须由服务端执行权限校验。

## 17. 成功标准

首期只有同时满足以下条件才算完成：

1. 170 属性黄金 DTS 满足全部结构和身份断言。
2. 源码树、有效树和解析来源链一致。
3. 身份歧义、配置集不完整、规格未审核和工具不可用都会阻止生产发布。
4. 导出的项目配置通过仓库集成的真实工具链。
5. 历史数据全部重映射，并保留不可变的原路径和原值证据。
6. JSON 和人工参数使用统一身份根，但不宣称具备未实现的解析能力。
7. 生产业务不再读写旧扁平参数身份模型。

## 18. 已收敛范围

本规格没有未决设计项。具体数据库字段、API 路由、实现文件边界、迁移命令顺序和任务级验证命令，在书面规格获得批准后由实施计划确定。
